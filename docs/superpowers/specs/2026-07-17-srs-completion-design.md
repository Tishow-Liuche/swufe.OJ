# Online OJ SRS Completion Design

## Goal

将《在线OJ（Online Judge）系统开发》需求规格说明书补全为一份面向长期教学、学习与竞赛使用的目标需求文档。文档只描述系统应提供的用户价值、业务规则和质量要求，不标注当前实现状态，也不包含技术架构、接口字段或代码细节。

## Scope

补全后的 SRS 以学生学习闭环为主线：

    发现题目
      -> 制定学习目标与计划
      -> 在线练习与自测
      -> 正式提交与获得反馈
      -> 收藏与错题复习
      -> 阅读或发布题解并参与社区交流
      -> 在课程、作业和比赛中应用
      -> 查看统计并调整后续学习

教师、管理员分别围绕教学组织、内容治理和平台运营展开需求。平台聚焦学习、教学与竞赛训练，不纳入广告、招聘、销售、会员推广或其他交易功能。

## Source Inventory

需求补全以现有系统的用户可见能力为参考，主要包括：

- 学习中心：学习仪表盘、每日练习、学习计划、计划签到、收藏、错题本；
- 题库与题单：公开题单、题单维护、题目筛选、题目详情、外部平台题目入口；
- 评测与记录：代码提交、提交详情、重测、个人统计、公开与范围排行榜；
- 教学与班级：学生加入班级、邀请码审核、班级成员、作业、作业报告与教师创建比赛；
- 比赛：报名、虚拟比赛、比赛提交、排名与排名快照；
- 社区：公告、帖子、回复、反应、题目反馈、举报、审核、通知与题解入口；
- 平台服务：外部 OJ 账号绑定、Codeforces/洛谷/QOJ/AtCoder 的辅助或同步能力、密码管理、管理员用户与班级管理。

这些能力将作为完整目标需求的依据，而不会在 SRS 中标注为“已实现”。

## Document Structure

保留现有的“引言—总体需求—用户需求与权限—功能需求—数据需求—非功能需求—业务规则—验收需求”骨架，并补强以下部分：

1. 总体需求：明确平台是学习型、教学型、竞赛型 OJ，禁止商业化和招聘类功能。
2. 用户需求与权限：完善学生、教师、管理员在学习、社区、教学和竞赛中的目标与边界。
3. 功能需求：
   - 学习成长：每日练习、学习目标、学习计划、进度、错题、收藏、复习、推荐；
   - 题库训练：筛选、完成状态、题单、自测、题目反馈、题目版本与来源；
   - 评测反馈：多语言提交、状态、测试点摘要、申诉、重测、提交保护；
   - 社区题解：分层防剧透题解、帖子、回复、反应、举报、审核、通知；
   - 教学比赛：课程、班级邀请码、作业评分与补交、教师反馈、数据导出、正式与虚拟比赛；
   - 平台支持：帮助中心、诚信辅助、可访问性和外部平台合规入口。
4. 数据与质量需求：补充学习进度、通知、社区内容、作业成绩、比赛快照与审计的保密、可追溯和一致性规则。
5. 验收需求：用可观察的用户流程验证各类需求，而不以接口或内部技术实现作为验收表述。

## Consistency Rules

- 每条功能统一采用“系统应……”或“用户应能够……”的需求语气。
- 不出现“当前尚未实现”“后续预留”“第一阶段”等开发进度表述。
- 不出现框架、数据库、端口、接口 URL、数据表字段、类名或算法实现。
- 题解、讨论、通知、帮助中心等内容必须以学习为目的，并遵守剧透、内容审核和学术诚信规则。
- 任何个人代码、隐藏测试数据、认证凭据和敏感教学数据均遵循最小必要访问原则。
