## 1. 系统现状与功能范围

### 1.1 当前已经具备的基础

当前仓库已经具备以下基础代码：

- Vue 3 前端项目和 NestJS 后端项目；
- PostgreSQL、Prisma 数据模型；
- Redis 与 BullMQ 判题队列配置；
- 用户注册、登录、刷新令牌和退出接口；
- 题目基础 CRUD、搜索和批量导入接口；
- 学生代码提交、提交记录和提交详情接口；
- C、C++、Python、Java 的基础编译运行代码；
- 个人资料、提交统计、语言分布、难度分布和提交热力图；
- PostgreSQL、Redis、MinIO、go-judge 的 Docker Compose 基础配置；
- 前后端 TypeScript 构建通过，Prisma Schema 校验通过。

### 1.2 当前核心问题

当前系统还不能作为完整 OJ 验收或公开部署，主要原因如下：

1. 所有题目实际共用三组 A+B 测试数据，不能按题目正确判题；
2. 判题任务实际调用 `NativeJudgeService`，直接在业务服务器运行学生代码；
3. go-judge 客户端已经编写，但没有接入当前 Judge Processor；
4. 测试用例没有真正保存到数据库，导入文件中的测试数据会被丢弃；
5. 学生、教师、管理员之间没有真正的接口权限隔离；
6. 提交列表和详情可以匿名访问，存在源码和隐藏数据泄露；
7. 教师后台、排行榜、讨论、公告、监控和报表尚未实现；
8. Docker Compose 没有包含前端和后端应用，不能一键启动完整系统；
9. 自动化测试只有一个与 OJ 业务无关的 Hello World 测试；
10. README 对完成情况的描述明显超前，需要按照真实状态修正。

### 1.3 本期必须覆盖的业务范围

两周版本应完成以下业务闭环：

```text
教师创建题目
  -> 配置样例与隐藏测试用例
  -> 发布题目
  -> 学生浏览题目
  -> 在线编写并提交代码
  -> 判题任务进入队列
  -> go-judge 安全编译运行
  -> 系统聚合测试点结果
  -> 学生查看结果和历史记录
  -> 教师查看学生提交和基础统计
```

### 1.4 本期暂不纳入范围

以下内容不属于当前两周版本的必要交付：

- Remote Judge 和第三方 OJ 账号池；
- 洛谷、Codeforces 等第三方平台自动提交；
- ACM、IOI 竞赛系统和虚拟比赛；
- 完整课程、班级、作业系统；
- 自动代码查重算法；
- Special Judge、交互题、多文件题；
- AI 推荐、能力画像和自动学习规划；
- Prometheus、Grafana、Loki、Sentry 全套监控；
- Kubernetes 和多节点判题集群。

## 2. 用户角色待完成内容

### 2.1 游客

#### 待完成功能

- 可以访问首页、公告、公开题库和公开题目；
- 可以注册和登录；
- 访问学生或教师页面时跳转到登录页；

匿名请求提交详情、题目管理和用户管理接口时，后端必须返回 `401 Unauthorized`，不能只依靠前端隐藏菜单。

- 不允许管理测试用例；
- 不允许查看其他学生的源码；

- 不允许修改角色、账号状态等敏感字段；

## 3. 账号与认证功能

### 3.1 角色与权限控制

当前题目新增、导入、修改、发布和删除接口只校验是否登录，没有校验教师或管理员角色。

#### 待完成内容

- 增加统一的角色装饰器与角色守卫；
- 在后端接口上明确标记允许访问的角色；
- 学生接口、教师接口和管理员接口分别进行权限验证；
- 前端增加登录守卫和角色路由守卫；
- 教师或管理员菜单根据真实用户角色展示；
- 权限校验必须以后端为准，前端隐藏按钮不能代替后端权限控制。

### 3.2 个人资料安全

当前个人资料接口将请求体直接交给 Prisma，存在提交额外字段修改角色的风险。

#### 待完成内容

- 创建 `UpdateProfileDto`；
- 只允许修改昵称和头像等白名单字段；
- 明确拒绝 `role`、`status`、`password` 等额外字段；
- 记录用户资料修改时间；
- 前端增加个人资料编辑表单和保存反馈。

### 3.3 Token 管理

#### 待完成内容

- 退出登录时调用后端退出接口并清除当前会话；
- 修复登录后导航栏状态不立即更新的问题；
- 处理刷新令牌失败后的统一跳转；
- 生产环境优先将刷新令牌放入 `HttpOnly` Cookie；
- 不在日志、错误响应和前端页面中输出令牌；
- 为登录接口增加基础频率限制。

### 3.4 演示账号初始化

#### 待完成内容

- Seed 中创建管理员、教师和学生演示账号；
- 演示密码只用于开发环境；
- 生产部署首次启动时强制更换默认密码；
- README 中的演示账号必须与 Seed 实际数据一致。

## 4. 题库与测试数据管理

### 4.1 题目基础信息

#### 待完成内容

- 完善题目标题、描述、输入格式、输出格式、样例输入、样例输出和提示；
- 将难度统一为入门、简单、中等、困难四个层级，或提供明确映射；
- 保存时间限制、内存限制、输出限制和允许语言；
- 支持草稿、已发布和已下架状态；
- 学生只能查看已发布题目；
- 教师可以查询自己有权管理的草稿和已下架题目；
- 对题目状态值进行枚举校验，禁止任意字符串写入；
- 题目删除优先使用逻辑删除，避免破坏历史提交。

### 4.2 测试用例数据模型

当前 Schema 只有测试组计数，没有保存输入和标准输出的真实测试用例实体。

#### 待完成内容

- 新增 `ProblemTestCase` 或等价数据模型；
- 每个测试用例至少包含以下字段：

```text
id
problemVersionId
input
expectedOutput
score
order
isSample
createdAt
updatedAt
```

- 样例测试可以向学生公开；
- 隐藏测试只能由教师、管理员和判题 Worker 读取；
- 支持新增、修改、删除和调整测试用例顺序；
- 删除题目版本时正确处理测试用例关联；
- 两周版本使用数据库保存小型文本测试用例，不强制接入 MinIO。

### 4.3 题目导入

当前导入页携带测试数据，但后端没有保存这些数据。

#### 待完成内容

- 定义固定的 JSON 导题格式；
- 使用 DTO 校验每道题的字段和测试用例；
- 导入题面、标签、样例和隐藏测试数据；
- 对重复标题或外部编号给出明确处理结果；
- 导入过程使用事务，单题失败不能留下不完整数据；
- 返回新增、跳过和失败的逐题结果；
- 不再在前端硬编码大量题目作为正式题库来源。

### 4.4 题面安全展示

#### 待完成内容

- 选择 Markdown 渲染器或经过清洗的 HTML 渲染方案；
- 禁止直接渲染未经净化的 `v-html`；
- 移除脚本、事件处理器、危险 URL 和内联注入；
- 正确展示代码块、表格和换行；
- 数学公式支持属于建议项，不阻塞核心验收。

## 5. 本地评测功能

### 5.1 go-judge 接入

当前 Judge Processor 固定注入 `NativeJudgeService`，go-judge 只存在于未使用的客户端和 Docker 配置中。

#### 待完成内容

- Judge Processor 改为调用 `JudgeService`；
- 在纯 Linux 环境部署并验证 go-judge；
- go-judge 仅允许后端或 Worker 内网访问，不向公网开放；
- 业务后端不再直接执行学生源码；
- 禁用或明确限制 `NativeJudgeService` 只能用于受信任的本地调试；
- 为 go-judge 请求设置 CPU、内存、进程数、输出和运行时间限制；
- 判题完成后清理缓存文件和临时数据；
- go-judge 不可用时返回 `SYSTEM_ERROR`，不能把任务永久留在判题中。

### 5.2 真实测试用例加载

当前所有题目固定运行三组 A+B 测试数据。

#### 待完成内容

- 根据 `problemVersionId` 加载对应测试用例；
- 没有测试用例的题目禁止发布或提交；
- 按 `order` 顺序执行测试用例；
- 保存每个测试点的状态、时间和内存；
- 学生响应中隐藏输入与标准输出；
- 教师响应可以在授权范围内查看测试数据；
- 重判前清理旧测试点结果，避免重复记录。

### 5.3 多语言支持

#### 必须支持

- C++17；
- Java；
- Python 3。

#### 待完成内容

- 修复 Java 源文件名和 `public class Main` 的匹配问题；
- 明确每种语言的版本、扩展名、编译命令和运行命令；
- 后端校验题目允许语言和用户提交语言；
- 限制源码长度；
- 为每种语言准备正确、编译错误、运行错误和超时样例；
- C 语言可作为额外支持，不阻塞三语言验收。

### 5.4 判题状态

系统至少需要统一支持：

```text
PENDING
QUEUING
COMPILING
RUNNING
ACCEPTED
WRONG_ANSWER
COMPILE_ERROR
TIME_LIMIT_EXCEEDED
MEMORY_LIMIT_EXCEEDED
RUNTIME_ERROR
OUTPUT_LIMIT_EXCEEDED
SYSTEM_ERROR
```

#### 待完成内容

- 定义统一枚举，不在前后端分别硬编码不同列表；
- 修复实际 WA 时测试点仍保存为 ACCEPTED 的问题；
- 修复失败测试点仍可能被计分的问题；
- 正确计算最终得分、最大时间和最大内存；
- TLE 必须严格使用题目时间限制，不能默认额外放宽 5 秒；
- MLE 必须来自真实内存限制和统计，不能始终为 0；
- 编译错误保存经过长度限制的编译器输出；
- 系统异常与用户代码错误必须分开。

### 5.5 队列可靠性

#### 待完成内容

- 使用稳定的 Job ID 防止重复入队；
- 创建提交、JudgeTask 和入队过程保证一致性；
- 失败重试不能重复创建测试点结果；
- 保存任务开始时间、结束时间、重试次数和错误原因；
- 支持管理员查看等待中、运行中和失败任务；
- 服务重启后未完成任务可以恢复或明确失败；
- 对单用户提交频率和并发任务数进行限制。

## 14. 安全与业务规则

### 14.1 提交与源码访问

- 匿名用户不能访问提交列表和详情；
- 学生只能查看自己的完整提交；
- 教师只查看其权限范围内的学生提交；
- 列表接口不返回完整源码；
- 隐藏测试输入和标准输出不得返回学生端；
- 错误日志删除服务器绝对路径和敏感环境信息。

### 14.2 提交限制

平台至少支持：

- 单用户每分钟提交次数限制；
- 单用户并发判题任务数限制；
- 单次源码长度限制；
- 单题连续提交间隔；
- 被教师限制账号的禁止提交状态；
- 超出限制时返回明确错误信息。

### 14.3 代码运行安全

- 学生代码只能在 go-judge 沙箱中运行；
- 禁止学生代码访问外网；
- 限制 CPU、内存、进程数、输出和运行时间；
- 测试死循环、内存耗尽、大量输出和创建子进程；
- go-judge 服务不持有数据库凭据；
- 判题失败不能影响业务 API 和数据库服务。

### 14.4 Web 安全

- 对题面、公告和讨论内容进行 XSS 清洗；
- DTO 拒绝额外字段；
- 状态、语言、难度和角色使用枚举白名单；
- 密码使用 bcrypt 等安全哈希；
- 生产 JWT 密钥必须随机生成；
- 不提交 `.env`、令牌和真实密码；
- 登录、注册、发帖和提交接口增加基础限流。

## 结论

当前项目最有价值的部分是前后端骨架、认证基础、题库接口、提交数据结构和个人统计。接下来的开发重点不是继续扩展功能数量，而是把以下四条主线做完整：

1. 真实测试用例；
2. 安全可靠的 go-judge 判题；
3. 学生、教师和管理员权限隔离；
4. 教师建题到学生提交结果的完整业务闭环。

完成第一优先级后，项目才具备实训验收的基本条件。排行榜、公告、讨论、监控和报表应在核心判题链路稳定后补充，Remote Judge、竞赛和智能推荐统一延期。
