## 3. 上线前检查

连接后先确认机器、存量服务和端口，不做破坏性操作：

```bash
hostnamectl
whoami
pwd
df -h
free -h
sudo systemctl --type=service --state=running
sudo docker ps -a
sudo ss -ltnp
```

如需停用旧任务，先记录服务或容器名称、镜像、卷和当前日志，再只停止已确认属于旧任务的对象：

```bash
sudo docker logs --tail 100 <container-name>
sudo docker inspect <container-name>
sudo docker stop <container-name>
```

不要执行 `docker system prune -a --volumes`，不要删除 `postgres_data`、`redis_data` 或 `minio_data` 卷。

## 4. 网络与域名

部署前必须满足以下条件：

1. DuckDNS 中 `subswufe.duckdns.org` 的 A 记录解析到 `20.205.119.164`。
2. Azure NSG 允许 TCP `80` 和 `443` 入站；SSH `22` 应限制为运维人员可信来源。
3. 主机防火墙放行 SSH、HTTP、HTTPS。若启用 UFW，先放行 SSH 再启用：

```bash
sudo ufw allow OpenSSH
sudo ufw allow 80/tcp
sudo ufw allow 443/tcp
sudo ufw enable
sudo ufw status verbose
```

4. 在服务器上确认解析结果：

```bash
getent hosts subswufe.duckdns.org
```

Caddy 会在域名、80/443 入站均就绪后自动申请 HTTPS 证书。不要用裸 IP 代替域名启动正式代理。

## 5. 安装运行环境

若机器尚未安装 Git 和 Docker，先安装并验证。若已存在 Docker，保留现有版本，先执行验证命令即可；不要在有运行中容器的机器上盲目移除或替换 Docker 软件包。

```bash
sudo apt update
sudo apt install -y ca-certificates curl git
sudo install -m 0755 -d /etc/apt/keyrings
sudo curl -fsSL https://download.docker.com/linux/ubuntu/gpg -o /etc/apt/keyrings/docker.asc
sudo chmod a+r /etc/apt/keyrings/docker.asc
sudo tee /etc/apt/sources.list.d/docker.sources >/dev/null <<EOF
Types: deb
URIs: https://download.docker.com/linux/ubuntu
Suites: $(. /etc/os-release && echo "${UBUNTU_CODENAME:-$VERSION_CODENAME}")
Components: stable
Architectures: $(dpkg --print-architecture)
Signed-By: /etc/apt/keyrings/docker.asc
EOF
sudo apt update
sudo apt install -y docker-ce docker-ce-cli containerd.io docker-buildx-plugin docker-compose-plugin
sudo systemctl enable --now docker
sudo usermod -aG docker zambo
docker --version
docker compose version
```

执行 `usermod` 后，重新登录 SSH 会话使组权限生效。Docker 官方 Ubuntu 安装说明见 <https://docs.docker.com/engine/install/ubuntu/>。

## 6. 获取代码与私有配置

建议部署目录为 `/srv/swufe-oj`。将 `<repository-url>` 替换为实际 Git 远端地址：

```bash
sudo install -d -o zambo -g zambo /srv/swufe-oj
git clone <repository-url> /srv/swufe-oj
cd /srv/swufe-oj
git fetch origin
git switch main
git pull --ff-only origin main
```

部署密钥文件必须仅保留在服务器上，并限制权限：

```bash
cp config/infra.env.example config/infra.env
cp config/app.prod.env.example config/app.prod.env
cp config/proxy.env.example config/proxy.env
chmod 600 config/infra.env config/app.prod.env config/proxy.env
```

配置要求：

- `config/infra.env`：为 PostgreSQL、Redis、MinIO 使用彼此不同的随机强密钥。
- `config/app.prod.env`：`DATABASE_URL`、`REDIS_PASSWORD`、MinIO 密钥必须与 `infra.env` 对齐；`JWT_ACCESS_SECRET` 和 `JWT_REFRESH_SECRET` 使用新的随机值。
- `config/proxy.env`：设置 `OJ_DOMAIN=subswufe.duckdns.org`。
- 不要将上述三个真实配置文件加入 Git。

项目脚本也会在缺少 `infra.env`、`app.prod.env` 时生成随机配置。首次部署可直接使用：

```bash
cd /srv/swufe-oj
export OJ_DOMAIN=subswufe.duckdns.org
bash scripts/deploy-production.sh
```

## 7. 部署、迁移与验证

部署脚本会构建前后端镜像、启动 Compose 服务，并在后端容器启动前执行 `prisma migrate deploy`。数据库迁移前先备份：

```bash
cd /srv/swufe-oj
mkdir -p backups
sudo docker exec oj-postgres sh -c 'pg_dump -U "$POSTGRES_USER" "$POSTGRES_DB"' \
  > "backups/oj-$(date +%F-%H%M%S).sql"
```

如需手动检查，使用统一 Compose 命令：

```bash
cd /srv/swufe-oj
compose() {
  sudo docker compose -f docker-compose.yml -f docker-compose.prod.yml \
    --env-file config/infra.env "$@"
}

compose config
compose up -d --build
compose ps
compose logs --tail 120 backend
compose exec -T backend npx prisma migrate status --schema=prisma/schema.prisma
```

服务就绪后验证：

```bash
compose exec -T backend wget -q --spider http://127.0.0.1:3000/
curl -fsSI https://subswufe.duckdns.org/
curl -fsS https://subswufe.duckdns.org/api/stats
```

浏览器验收至少覆盖：注册、登录、题库搜索、题目详情、一次本地提交、管理员用户管理页。若本次代码包含数据库迁移，确认 `compose exec -T backend npx prisma migrate status --schema=prisma/schema.prisma` 显示迁移一致。

## 8. 后续更新与回滚

常规更新：

```bash
cd /srv/swufe-oj
git fetch origin
git switch main
git pull --ff-only origin main
bash scripts/deploy-production.sh
```

代码回滚前先恢复到已验证提交并重新构建；数据库迁移不可假定可逆，涉及数据结构变更时优先从第 7 节的备份恢复，而不是使用 `prisma db push --accept-data-loss`：

```bash
git log --oneline -10
git switch --detach <known-good-commit>
bash scripts/deploy-production.sh
```

完成回滚验证后，再决定是否将 `main` 前移或创建修复提交。

## 9. 故障定位顺序

```bash
cd /srv/swufe-oj
compose ps
compose logs --tail 200 backend
compose logs --tail 100 caddy
compose logs --tail 100 postgres
compose logs --tail 100 redis
compose logs --tail 100 go-judge
```

优先确认域名解析、80/443 入站、Caddy 证书、后端健康检查和 Prisma 迁移状态。不要为排障向公网开放 PostgreSQL、Redis、MinIO 或 go-judge 端口。
